Overview

What the Accelerator Does

The Agentic GenAI Security Accelerator ingests AWS security findings, normalizes and deduplicates results, maps findings to security pillars, calculates an explainable 0–5 maturity view, uses Amazon Bedrock for security reasoning, supports read-only AWS investigation, generates remediation plans, routes actions through approval, runs dry-run execution by default, and records audit evidence.

It is designed for security teams who want to move from scattered findings to prioritized, measurable, and governed security improvement — without giving AI unrestricted access to AWS resources.

Get Started

Try It With Sample Findings

The repository includes sample Prowler-style findings so you can experience the full workflow without connecting an AWS account first. 

git clone https://github.com/lxrsd/agentic-genai-security-accelerator.git
cd agentic-genai-security-accelerator
./scripts/setup_demo.sh
./scripts/run_demo.sh

Then open http://127.0.0.1:8080 in your browser.

What You'll See

  • Sample findings load automatically
  • Security maturity score is populated
  • Pillars show evaluated results
  • Grouped findings with deduplication
  • AI assistant available (requires Bedrock)
  • Remediation planning and queue
  • Dry-run execution workflow
  • Audit log records actions

Default Mode

  • Investigation: enabled
  • Planning: enabled
  • Approval: required
  • Dry-run execution: enabled
  • Live AWS changes: disabled
  • Medium/high-risk remediation: disabled

No AWS resources are modified in this mode.

Setup

What Gets Installed Automatically

setup_demo.sh prepares the local environment:

  • Creates a local Python virtual environment (.venv)
  • Installs dependencies from requirements.txt
  • Installs or validates Prowler if needed
  • Creates .env from .env.demo
  • Creates mcp_config.json from the example config
  • Runs a preflight check
  • Verifies whether AWS CLI and Bedrock are available

You may see terminal prompts depending on your system. These are part of local setup and do not enable live remediation.

AI Assistant

Amazon Bedrock Access

Amazon Bedrock is used for the AI assistant and security reasoning. If Bedrock model access or SDK support is missing, the dashboard still runs — but AI chat will be unavailable until Bedrock is configured.

  • Bedrock usage incurs token-based AWS charges
  • Model access must be enabled in the selected AWS region
  • Updated boto3/botocore (≥1.34) is required for Bedrock Runtime
  • Without Bedrock, the sample workflow and scoring still function

Safety

Safe by Default

The default local mode runs the full workflow in dry-run:

INVESTIGATION_TOOLS_ENABLED=true
REMEDIATION_PLANNING_ENABLED=true
REMEDIATION_EXECUTION_ENABLED=true
DRY_RUN_REMEDIATION=true
REQUIRE_APPROVAL_FOR_ALL_REMEDIATION=true
ALLOW_MEDIUM_RISK_REMEDIATION=false
ALLOW_HIGH_RISK_REMEDIATION=false
Investigation enabled Planning enabled Approval required Dry-run execution enabled Live AWS changes disabled Medium/high-risk disabled

Remediation

Live Remediation Is Explicitly Opt-In

Connecting AWS does not enable live remediation. Live remediation requires all of:

  • DRY_RUN_REMEDIATION=false
  • EXECUTION_ROLE_ARN configured
  • Approved remediation action ID
  • Low-risk allowlisted action
  • Before/after verification
  • Audit log entry

Live remediation should be tested only in a sandbox or controlled AWS account. The recommended starting point is the sample or dry-run workflow.

Cost

Cost Model: Usage-Based and Controlled

  • Sample findings and local walkthrough can run without AWS usage
  • Bedrock usage is token-based if AI chat is used
  • S3 report storage is usually small for workshop-scale usage
  • Connected AWS scans vary by account size and enabled services
  • Live remediation is optional and may create service-specific charges

The recommended starting point is the sample or dry-run workflow. Customers can decide later whether to connect AWS, enable additional security service context, or test low-risk live remediation.

Help

Troubleshooting

  • .venv not found: Run ./scripts/setup_demo.sh
  • Port 8080 busy: Stop old server or kill -9 $(lsof -ti:8080)
  • Sample findings show 0: Confirm sample-data/prowler-output/sample-findings.json exists
  • Bedrock unavailable: Confirm model access, AWS region, and boto3 ≥ 1.34
  • AWS not connected: Configure AWS CLI or use sample data first
